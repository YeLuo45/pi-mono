## Context

当前 Studio 以“园区沙盘 + 招聘/派单 + 事件回放”为核心，已经能把 LLM 输出与工作流映射为可视化事件；并且已有：

- Provider 配置（local/cloud）与连通测试
- 任务队列（enqueue/runJob）与按 projectId 的预览输出
- 策略面板（policy）与部门工单入口

但缺少一个“公司治理层”的入口：老板的日常指挥往往是跨部门的、会改变方向的；如果没有可归档的章程与变更控制，系统会退化为随机聊天与重复返工，难以形成“经营感”和“流程学习”。

本变更引入“会议室（立项/变更会）→ 章程（目标/里程碑/节点）→ 归档 → 偏离提醒（秘书）→ 变更确认并归档新版本”的闭环，并将模型路由做成“游戏化简单配置”：执行默认小模型/本地，方向决策默认强模型/云端。

## Goals / Non-Goals

**Goals:**

- 在 Web 场景中新增会议室入口（放置在策划区下方空地），点击打开“立项/变更会”弹窗。
- 为每个项目提供“章程（Charter）”数据模型：`goal`、`milestones[]`、`nodes[]`、`version`、`archivedAt?`。
- 提供服务端 API：读取/保存当前草稿章程；归档章程（生成新版本）；读取历史版本（最小实现）。
- 提供偏离检测：已归档版本存在时，若草稿章程相对归档版发生变化（goal/milestones/nodes 任一变化），生成变更记录并通过秘书提示（事件/通知）。
- 提供模型路由的最小规则与 UI：会议默认使用 cloud；三总监默认跟随会议；其他岗位默认 local；个体覆盖仅在高级入口出现。

**Non-Goals:**

- 不实现复杂的多人权限/审批流；默认单人本地运行。
- 不实现完整的依赖图与跨部门契约验证（先以 goal/milestones/nodes 的“偏离提醒”代替）。
- 不强制阻止玩家在执行阶段改方向；只做软提醒与变更归档引导。
- 不要求会议必须调用强推理大模型；规则模式可用，LLM 增强作为可选升级。

## Decisions

- **章程字段最小化**：只锁定 `目标/里程碑/节点` 三类核心字段。
  - 替代方案：加入预算/人力/资产清单等；MVP 先不做，避免配置爆炸。
- **版本与归档语义**：项目维持一个“当前草稿”与一个“最新归档版本”，归档时 `version+1` 并写入历史列表。
  - 替代方案：像 git 一样多分支；MVP 不需要。
- **偏离检测只比较三类字段**：goal 文本、milestones 列表、nodes 列表（顺序与内容）。
  - 替代方案：比较更细粒度状态（如验收点、文件变更）；后续迭代可扩展。
- **会议模型路由采用“三级默认 + 少量覆盖”**：
  - 全局：执行/会议模型选择（local/cloud/auto 三档）
  - 角色：三总监默认跟随会议
  - 个体：默认跟随角色（高级覆盖）
  - 替代方案：每个 agent 配完整 baseUrl/model/apiKey；MVP 不可用（操作复杂且容易误配）。
- **秘书提示为第一等公民**：偏离检测、归档成功、变更待确认等都以通知/事件形式记录，便于回放与“经营学习”。

## Risks / Trade-offs

- [Risk] 章程过于简化导致“看起来像流程但不指导执行” → Mitigation：节点（nodes）鼓励写成可检查的验收点，并在部门工单模板中引用。
- [Risk] 日常指挥变成聊天噪音 → Mitigation：部门对话最终落成工单；偏离提醒把“方向变化”推回会议室确认并归档。
- [Risk] 模型选择仍会让玩家困惑 → Mitigation：用三档按钮（省钱/均衡/高质量）与“会议默认强模型”文案引导；细节只在高级入口出现。
- [Risk] 偏离提醒过于频繁 → Mitigation：节流（例如 30s 内合并提醒）、只对已归档版本触发、只比较三类字段。
